# Day 009 — PipeTwin

Date: 2026-06-27
Status: blocked

## One-line pitch

A developer CLI that turns a failed CI run URL or pipeline job into a local repro bundle: checked-out commit, runner image hints, service stubs, env diff, and a single command teammates can run before pushing another blind fix.

## Problem

CI failures still waste developer time because the failing environment is often outside the developer’s normal edit-run loop. Teams push speculative fixes, wait for hosted runners, scan logs, guess which env vars or services mattered, and repeat. Existing local-CI tools prove the need, but they either target one CI dialect, ask teams to adopt a new pipeline system, or require the developer to already know how to map a hosted failure into a local command.

The status-quo pain clears the threshold: it can block releases, burn more than 30 minutes per failure, and create noisy commit histories or embarrassing “fix CI again” churn in public repos.

## Target user

Small teams and solo maintainers who:

- use GitHub Actions, GitLab CI, Woodpecker/Drone-style YAML, PikoCI, or Concourse-like pipelines,
- maintain open-source or SaaS repos where CI failures block merges/releases,
- do not want to replace their CI platform just to get local repro,
- are comfortable running a CLI with Docker or Podman,
- need a shareable repro command for teammates or contributors.

## MVP scope

- CLI command: `pipetwin capture <failed-run-url-or-log-file>`.
- GitHub Actions first-class path for v0.1: fetch workflow/job metadata when public or pasted, identify commit SHA, job name, matrix values, container image hints, env keys, and service containers.
- Generate `pipetwin.repro.yml` plus a shell command that checks out the right commit, starts service stubs, and runs the closest local job command.
- Secret-safe placeholders: list missing env keys without storing values.
- `pipetwin explain` report: what was reproduced exactly, what was approximated, and what still must be checked in hosted CI.
- Markdown share output for PR comments or issue replies.

Non-goals for v0.1: full CI replacement, perfect emulation of every hosted runner feature, secret sync, private API dependence, or cloud execution.

## Competitor / Substitute Check

| Type | Name / Substitute | Notes |
|---|---|---|
| Direct competitor | nektos/act | Strong for GitHub Actions local execution and a direct substitute for part of the job. PipeTwin’s wedge is starting from a failed run/log and producing a repro bundle with env/service gaps, not only running a chosen workflow locally. |
| Direct competitor | Dagger | Powerful local-first CI/runtime with broad language SDKs and observability. It requires adopting Dagger modules; PipeTwin should sit beside the CI a team already has. |
| Direct competitor | PikoCI / Concourse local execution | Both validate local job execution and service-aware pipelines. They are strongest inside their own pipeline model; PipeTwin targets teams that are not ready to migrate CI. |
| Direct competitor | Earthly / Taskfile / Makefile / Dev Containers | Good reproducible-task building blocks. They do not automatically translate a hosted CI failure into a teammate-shareable local repro. |
| Indirect substitute | Push another commit, re-run hosted CI, read logs manually | Lowest setup cost, but wastes cycles and is publicly noisy for open-source repos. |
| Indirect substitute | One-off Docker Compose file or shell script | Works when the maintainer already understands the failure environment; hard to keep aligned with CI YAML and matrix jobs. |
| Status quo | Debug in the hosted runner by trial-and-error | Tolerable for rare failures, painful for flaky services, matrix jobs, or contributors without access to private runner context. |

## Wedge

**Wedge-first line:** maintainers debugging recurring CI failures in small public/private repos → `act`, Dagger, PikoCI/Concourse local execution, Docker Compose, Makefiles, and hosted reruns → substitutes either require choosing the right job manually, adopting a new CI model, or already knowing which env/services caused the failure → a “failed run/log → local repro bundle” CLI that reports exact vs approximated runner state and produces a shareable command → GitHub Actions/local-CI search content, r/webdev/selfhosted CI threads, and replies to “why does CI fail only remotely?” discussions → fresh local-first CI projects are highlighting “no push, no waiting” as the core pain.

PipeTwin should not compete as yet another CI platform. It should become the missing adapter between hosted failures and local debugging.

## Kill condition

Reject or narrow if `act` plus a short README already solves the painful job for the target repos, or if reliable repro requires private CI APIs/secrets that cannot be accessed safely in a public-safe MVP. Also narrow if users want a full pipeline engine; that moves PipeTwin into a crowded CI replacement market where PikoCI, Concourse, Dagger, GitHub Actions, GitLab, Woodpecker, Drone, Earthly, and Buildkite already dominate.

## Decision

Create the dedicated project scaffold for PipeTwin, but mark today as `blocked` rather than cleanly `repo-created` until the scaffold is pushed to GitHub. The idea clears the 18/25 threshold, the distribution gate (4/5), and the competitive-wedge gate (3/5). The blocker is operational, not product-quality: GitHub write access has previously failed for scaffold pushes.
